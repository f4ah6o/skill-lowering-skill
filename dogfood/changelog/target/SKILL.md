# changelog

このスキルは、一貫性があり変更の影響を把握できる `CHANGES.md` を作成するために使用する。

## 目的

`CHANGES.md` は、利用者に影響する変更を記録した履歴である。
利用者、運用者、保守担当者、将来作業するエージェントが、変更内容とその影響を差分を読まずに理解できるように書く。

## ファイル名

リポジトリですでに `CHANGELOG.md` を使用している場合を除き、`CHANGES.md` を使用する。
明示的に求められない限り、両方を作成しない。

## 構成

次の構成を使用する。

```markdown
# Changes

## Unreleased

### Added

### Changed

### Fixed

### Deprecated

### Removed

### Security

### Migration
```

リポジトリに異なる慣例がなければ、`Unreleased` 配下の空のセクションを残す。

## 項目の書き方

各項目は、利用者が理解できる具体的で短い文にする。
ローカルイシュー、PR、コミットがある場合はリンクを付ける。

例を次に示す。

```markdown
- スキルを使った開発向けに、ローカルイシューのテンプレートを追加した。(`issues/polished/20260622-local-issues.md`)
- 偽装されたドメインを拒否するように、Git ホストの検証を修正した。
- ローカルファイルを置き換える前に日時付きのバックアップを作成するように、スキルの更新手順を変更した。
```

## カテゴリの規則

- **`Added`**：新しい機能、コマンド、ファイル。
- **`Changed`**：動作や利用者体験の変更、利用者に影響するリファクタリング。
- **`Fixed`**：バグ修正。
- **`Deprecated`**：現在は対応しているが使用を推奨せず、将来削除する予定の動作。
- **`Removed`**：削除した機能や互換性対応。
- **`Security`**：堅牢化とセキュリティに関する修正。
- **`Migration`**：必要な手作業、破壊的変更、データ移行。

## 項目が必要な変更

公開 API、CLI の動作、生成ファイル、UI、設定、データ形式、インストール、アップグレード手順、互換性、堅牢化、移行に影響する変更には項目を追加する。

## 項目が不要な変更

内部の誤字だけを直す変更、利用者に影響しないテストだけの変更、動作に影響しないリファクタリング、リリースやインストールの動作に影響しない CI だけの変更には項目を追加しない。

## バージョンのリリース手順

リリースするときは、次の手順に従う。

1. `## Unreleased` を `## <version> - YYYY-MM-DD` に変更する。
2. その上に新しい空の `## Unreleased` セクションを挿入する。
3. 各項目が一つのカテゴリだけに属することを確認する。
4. 破壊的変更や手作業が必要な変更には移行時の注意事項を追加する。
5. リリース日は ISO 形式で記載する。

## レビュー用チェックリスト

変更を確定する前に、次の点を確認する。

- 影響を受ける利用者が差分を読まずに変更内容を理解できる。
- 堅牢化や移行への影響を一般的なカテゴリに紛れ込ませていない。
- 項目に認証情報やチャットにしかない文脈を含めていない。
- 実装の詳細だけを述べた不要な項目がない。
